üîê Why Authorization at NestJS Layer vs API Gateway?

This is a common architectural question. Here's why we handle authorization at the NestJS layer:

‚úÖ NestJS Authorization (Our Choice)
  • Business Context: Access to user/device data
  • Resource-based: Device ownership validation
  • Complex Rules: Multi-tenant permissions
  • Database Access: User roles & permissions
  • Flexibility: Custom authorization logic
  • Debugging: Easier to trace and debug
‚ùå Gateway Authorization (Alternative)
  • Limited Context: No business logic access
  • Simple Rules: Only basic token validation
  • No DB Access: Can't check ownership
  • Less Flexible: Hard to customize
  • Vendor Lock-in: Gateway-specific features
  • Best for: Simple API key validation

üéØ Hybrid Approach (Best Practice):

  • API Gateway: Basic authentication (API keys, rate limiting)
  • NestJS Layer: Authorization (permissions, resource access)
  • Result: Double security with appropriate separation of concerns

üîç Concrete Example: Device Goes Offline
